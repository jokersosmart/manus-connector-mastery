### B. 收斂 (五何法)

#### 元素：過去 - 自己 - Input data (其他類似連結器的開發經驗)

- **核心概念**：經驗的「可轉移性」
- **為什麼？** 因為不是所有經驗都能直接套用。
- **為什麼？** 因為不同平台的 API 設計哲學、限制和社群生態都不同。
- **為什麼？** 因為 Airtable 強調的是「賦能非開發者」，其 API 設計可能更注重易用性而非純粹的技術彈性，這與某些面向開發者的平台（如 AWS）不同。
- **為什麼？** 因為目標用戶不同，產品的核心價值主張也不同。Airtable 的成功在於其使用者體驗和靈活性，而不僅僅是技術能力。
- **根本原因**：**開發者必須從「賦能非開發者」的用戶視角出發，去理解和設計連結器，而不僅僅是將過去純技術導向的 API 串接經驗直接複製過來。**

#### 元素：過去 - 別人 - Input data (使用電子表格的經驗)

- **核心概念**：「電子表格」的思維慣性
- **為什麼？** 因為使用者會習慣用 Excel 或 Google Sheets 的方式來思考資料結構，例如單一的工作表、自由的儲存格格式。
- **為什麼？** 因為他們缺乏「關聯式資料庫」的概念，如主鍵、外鍵、資料表之間的連結。
- **為什麼？** 因為這是他們最熟悉、最直觀的工具，而「資料庫」聽起來很嚇人、很複雜。
- **為什麼？** 因為傳統資料庫需要專業知識來設計和維護，而電子表格幾乎沒有入門門檻。
- **根本原因**：**使用者最大的心智模型障礙是從「無結構的電子表格」過渡到「有結構的關聯式資料庫」。連結器的設計和教學必須要能彌平這個鴻溝。**

#### 元素：現在 - 自己 - Input control (決定要新增、修改或棄用哪些工具)

- **核心概念**：功能的「優先級」
- **為什麼？** 因為開發資源有限，不可能一次實現所有 Airtable API 的功能。
- **為什麼？** 因為必須根據使用者的核心需求來決定開發順序，即 80/20 法則。
- **為什麼？** 因為要找出哪些功能能為使用者創造最大的價值，解決他們最迫切的問題。
- **為什麼？** 因為這需要深入了解使用者的應用場景，例如他們最常用 Airtable 做什麼？是專案管理、CRM 還是內容規劃？
- **根本原因**：**連結器功能的開發優先級，必須基於對使用者核心應用場景的深刻理解，而非 API 功能的完整性。**

#### 元素：未來 - 別人 - Input control (決定是否付費升級 Airtable)

- **核心概念**：「升級」的決策動機
- **為什麼？** 因為免費版的功能或用量達到上限，無法滿足業務需求。
- **為什麼？** 因為業務成長了，需要更進階的功能，如更長的歷史紀錄、更複雜的自動化、更精細的權限控制。
- **為什麼？** 因為他們已經透過 Airtable 和 Manus 連結器看到了顯著的投資回報 (ROI)，證明了這個工具的價值。
- **為什麼？** 因為這個工具已經深度整合到他們的核心工作流程中，成為不可或缺的一部分。
- **根本原因**：**使用者付費升級的根本動機，源自於工具已成功證明其價值，並成為其業務成長的關鍵驅動力。因此，連結器的設計應專注於幫助使用者盡快實現這種價值。**

### 本輪根本原因清單

1.  **開發視角轉換**：開發者必須從「賦能非開發者」的用戶視角出發，去理解和設計連結器。
2.  **心智模型過渡**：連結器的設計和教學必須幫助使用者從「電子表格」過渡到「關聯式資料庫」。
3.  **需求驅動開發**：連結器功能的開發優先級，必須基於對使用者核心應用場景的深刻理解。
4.  **價值驅動升級**：連結器的設計應專注於幫助使用者盡快實現業務價值，以驅動其付費意願。
