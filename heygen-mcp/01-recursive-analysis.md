### B. 收斂 (五何法)

#### 元素 1: HeyGen API 的使用成本 (現在 - 自己 - Input data)

- **為什麼這是一個關鍵輸入？** 因為它直接影響連結器的開發和運營成本，並可能最終轉嫁給使用者。
- **為什麼會有這個成本？** 因為 HeyGen 作為一個商業服務，其 AI 模型運算、伺服器資源和研發投入需要透過 API 收費來變現。
- **為什麼我們需要使用這個 API？** 因為從頭開發同等級的 AI 影片生成技術，成本遠高於直接使用成熟的 API 服務。
- **為什麼成本結構是這樣設計的？** HeyGen 可能根據影片解析度、時長、使用的功能 (如虛擬人、聲音克隆) 等多維度來計費，以反映不同的運算資源消耗。
- **為什麼這對我們很重要？** **(根本原因)** 我們需要在連結器的設計中，提供清晰的成本預估或用量控制機制，讓使用者在調用前能意識到潛在費用，避免產生意外的高額帳單，這對於建立使用者信任至關重要。

#### 元素 2: 撰寫有效的文字提示 (現在 - 別人 - Input data)

- **為什麼使用者需要投入精力來寫提示？** 因為 AI 的輸出品質高度依賴輸入指令的清晰度和具體性。
- **為什麼這對使用者來說是個挑戰？** 因為大多數人習慣於圖形化介面操作，而非透過精確的自然語言來描述視覺畫面和動態效果。
- **為什麼不能讓 AI 自動理解模糊的指令？** 目前的 AI 模型還無法完全讀懂人類的潛在意圖，需要明確的指令來縮小解釋空間，確保生成結果符合預期。
- **為什麼提升提示技巧很重要？** 因為掌握了提示工程 (Prompt Engineering) 的使用者，能以更低的試錯成本，更高效地創造出更高品質的影片。
- **為什麼這對連結器設計有啟示？** **(根本原因)** 連結器不僅僅是功能的封裝，更應該扮演「智慧助理」的角色。我們應該設計引導式的指令範本、提供多樣化的使用案例，甚至在工具層級將複雜的影片設定拆解為更簡單、更易於理解的參數，以降低使用者的學習門檻。

### C. 遞迴

**本輪根本原因清單 (作為 Loop 2 的主題)：**

1.  **成本透明度與用量控制**：如何在連結器層級設計機制，幫助使用者管理 HeyGen API 的使用成本。
2.  **降低使用者學習門檻**：如何將複雜的影片生成功能，轉化為 Manus 中更直觀、更易於使用的工具和指令。

## 循環 Loop 2

**主題**：為 HeyGen 連結器設計成本控制機制與低門檻的使用者體驗

**思考模式**：平行式

### A. 發散 (4! 分析)

| 時間 | 元素 | 自己 (Manus 團隊) | 別人 (終端使用者) |
| :--- | :--- | :--- | :--- |
| **過去** | **Input data** | - Loop 1 的分析結果<br>- 其他有成本的 MCP 連結器 (如 OpenAI) 的設計經驗 | - 對其他付費軟體「用量不清」的負面經驗<br>- 因功能複雜而放棄學習某個工具的經歷 |
| | **Input control** | - 參考現有連結器，決定成本提示的互動方式 (例如，執行前確認) | - 依賴過去經驗，猜測哪些操作可能會產生費用 |
| | **Library** | - 一套用於計算和顯示 API 成本的通用模組<br>- 「引導式教學」的設計模式 | - 從其他工具學會的「先用免費版試水」的習慣 |
| | **Output** | - 初步的成本控制和使用者引導的設計草案 | - 對於新工具抱持「可能很貴、可能很難」的觀望態度 |
| **現在** | **Input data** | - 研發這些機制所需的前端與後端工作量<br>- 需要解析 HeyGen 的複雜定價模型 | - 閱讀文件和範例，理解成本如何計算以及如何下指令的時間 |

| | **Library** | - Manus 平台的 UI 元件庫 (用於顯示確認對話框)<br>- 參數化工具設計的最佳實踐 | - 使用者在社群分享的「省錢秘訣」或「好用指令」<br>- 對於「所見即所得」編輯器的偏好 |
| | **Output** | - 在連結器中實作的成本預警系統和多層次的工具集 | - 一個既能快速上手，又能精確控制成本和效果的影片生成體驗 |
| **未來** | **Input data** | - 監控成本預估的準確性並持續校準<br>- 收集使用者回饋，了解引導流程是否有效 | - 為更進階的功能或更高品質的輸出支付額外費用<br>- 貢獻自己的模板或使用訣竅到社群 |
| | **Input control** | - 決定是否要引入更智慧的成本優化建議 (例如，「使用這個聲音會比另一個便宜 20%」) | - 選擇是否要訂閱一個「無限量」方案 (如果 HeyGen 或 Manus 提供) |
| | **Library** | - 一個經過驗證的、適用於所有 MCP 連結器的成本管理與使用者引導框架 | - 一套個人化的、結合了成本效益和創作效率的 HeyGen 工作流程 |
| | **Output** | - HeyGen 連結器因其「使用者友善」和「成本透明」而獲得良好口碑 | - 使用者社群蓬勃發展，圍繞連結器產生大量 UGC (使用者生成內容) |

### B. 收斂 (五何法)

#### 元素 1: 設計工具的抽象層級 (現在 - 自己 - Input control)

- **為什麼我們需要決定這個？** 因為它直接影響了使用者的複雜度和靈活度之間的平衡。
- **為什麼會有「包山包海」和「細粒度」的選項？** 「包山包海」對新手友好，一個指令完成所有事；「細粒度」對專家友好，可以精確控制影片的每一個細節。
- **為什麼不能只提供一種？** 因為我們的目標使用者光譜很廣，從只想快速生成影片的新手，到需要高度客製化的專業內容創作者，單一模式無法滿足所有人。
- **為什麼這個選擇很重要？** 它定義了連結器的核心互動模式，也決定了我們的文件和教學內容要如何組織。
- **為什麼這對使用者體驗至關重要？** **(根本原因)** 使用者需要一條平滑的學習曲線。一個理想的系統應該允許使用者以一個極其簡單的指令入門 (例如，`從這段文字生成一個影片`)，然後在他們準備好時，能逐步揭示更進階的控制選項，而不是一開始就用數十個參數壓垮他們。這就是「漸進式揭示」(Progressive Disclosure) 的設計哲學。

### C. 遞迴

**本輪根本原因清單 (作為 Loop 3 的主題)：**

1.  **漸進式揭示 (Progressive Disclosure)**：如何設計一套從單一指令到多工具組合的互動模型，讓使用者可以平滑地從新手過渡到專家。

## 循環 Loop 3

**主題**：為 HeyGen 連結器設計「漸進式揭示」的互動模型

**思考模式**：序列式 (專注於 Manus 團隊的內部設計過程)

### A. 發散 (4! 分析)

| 時間 | 元素 | 自己 (Manus 團隊) |
| :--- | :--- | :--- |
| **過去** | **Input data** | - Loop 2 的分析結果<br>- 其他軟體 (如遊戲、專業設計工具) 的新手教學模式 | 
| | **Input control** | - 決定以「工具集」而非「單一工具」的模式來設計 | 
| | **Library** | - 軟體設計中「引導模式 (Wizard)」和「專家模式 (Expert Mode)」的概念 | 
| | **Output** | - 一個關於如何組織 HeyGen 功能的初步分類 | 
| **現在** | **Input data** | - 將 HeyGen 所有 API 功能點進行盤點和分類的工作量 | 
| | **Input control** | - **定義「新手」、「進階」、「專家」三個層級的使用者場景**<br>- 設計一套可以串連不同工具的「工作流 (Workflow)」概念 | 
| | **Library** | - Manus MCP 對 `tool-chaining` (工具鏈) 的底層支援 |

| | **Library** | - 一套衡量使用者從新手到專家轉化率的數據分析模型 | 
| | **Output** | - HeyGen 連結器成為 Manus 平台上「最易上手、功能也最強大」的典範 | 

### B. 收斂 (五何法)

#### 元素 1: 定義「新手」、「進階」、「專家」三個層級的使用者場景 (現在 - 自己 - Input control)

- **為什麼要定義這些層級？** 為了將複雜的功能集合，映射到不同使用者的真實需求和能力水平上。
- **為什麼是這三個層級？** 它們代表了典型的使用者成長路徑：從「我只想快速看到結果」到「我想對結果有一些控制」，再到「我需要完全的創作自由」。
- **為什麼這個定義如此關鍵？** 因為它將成為我們設計工具集、撰寫文件、甚至行銷推廣的核心框架。所有後續工作都將圍繞這三個層級展開。
- **為什麼我們需要為此投入精力？** 如果沒有清晰的用戶分層，我們可能會設計出一個對新手太複雜、對專家又太局限的「四不像」產品。
- **為什麼這能實現「漸進式揭示」？** **(根本原因)** 因為我們可以為每個層級設計專屬的、數量恰到好處的工具。新手只需要看到 1-2 個核心工具，進階使用者會看到 5-6 個，而專家則可以取用全部 10+ 個工具。使用者可以在自己的節奏下，從一個層級自然地探索到下一個層級，而不會感到不知所措。

### C. 遞迴

**本輪根本原因清單：**

1.  **使用者場景映射**：將抽象的功能，轉化為與使用者心智模型一致的、分層的工具集。

## 終止條件與核心洞察摘要

在 Loop 3 的分析中，`Input data` 和 `Input control` 欄位開始與 Loop 2 高度相似，新的洞察逐漸收斂。核心挑戰已經從「做什麼」轉向「如何設計細節」。因此，遞迴分析在第 3 層停止。

**核心洞察摘要：**

1.  **成本透明是信任的基石**：連結器必須提供內建的、清晰的成本預估和用量控制機制，這是讓使用者放心創作的前提。
2.  **降低門檻是推廣的關鍵**：必須將複雜的 API 功能，透過精心設計的抽象層和引導式體驗，轉化為新手友善的工具。不能假設使用者是專家。
3.  **漸進式揭示是成長的路徑**：應設計一個從簡單到複雜的多層次工具系統，讓使用者可以根據自己的需求和熟練度，平滑地從新手過渡到專家，實現「易上手，難精通」的理想體驗。
