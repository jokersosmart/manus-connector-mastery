# Jam 連結器：從新手到精通指南

**作者：Joker**

## 前言：改變偵錯的遊戲規則

歡迎來到 Jam 連結器的世界。本指南將帶您踏上一段從新手到精通的旅程，徹底改變您對軟體偵錯的看法和做法。傳統的偵錯充滿了來回溝通、資訊缺失和令人沮喪的「在我電腦上是好的啊」。Jam 的出現，就是為了解決這個核心痛點。

本指南將分為五個階段，每個階段都代表了您在使用 Jam 連結器時心態和技能的轉變。跟隨我們的腳步，您將不僅僅是學會一個工具，而是掌握一套全新的、高效的、由 AI 驅動的偵錯哲學。

---

## 第一階段：觀察者 (The Observer) - 理解「為什麼」

在成為一個高效的偵錯者之前，您必須先理解問題的本質。這個階段的目標是讓您明白 Jam 為何存在，以及它解決了什麼根本性的問題。

> **核心心態：** 從「又是一個 Bug 回報」轉變為「這是一個富含上下文的完整故事」。

### 基礎概念：什麼是「一個 Jam」？

想像一下，您不再需要閱讀使用者寫的冗長文字描述，也不再需要猜測他們到底點了什麼。一個「Jam」就是一個包含了所有偵錯所需資訊的數位包裹，它透過一個簡單的連結分享。

這個包裹裡有：

*   **螢幕錄影**：使用者操作的完整影片。
*   **使用者事件**：每一次點擊、輸入和頁面跳轉的精確記錄。
*   **主控台日誌**：所有前端的錯誤、警告和日誌訊息。
*   **網路請求**：每一個 API 呼叫的來龍去脈。
*   **環境資訊**：作業系統、瀏覽器版本等。
*   **自訂中繼資料**：開發者主動附加的使用者 ID、版本號等。

### 為何它如此重要？

*   **對回報者而言**：只需點擊錄製，重現問題，然後分享連結。無需撰寫複雜的報告。
*   **對開發者而言**：告別猜測。您擁有了一個可以完美重現問題的「數位時光機」，所有線索都已為您備妥。

**新手任務**：找一個您的同事，讓他錄製一個操作網站的 Jam 並把連結給您。您的任務不是解決問題，而是僅僅透過觀看 Jam 錄製檔，感受一下擁有完整上下文是什麼樣的體驗。

## 第二階段：分析師 (The Analyst) - 掌握「是什麼」

當您理解了 Jam 的價值後，下一步就是學會如何從這個資訊寶庫中提取有用的線索。這個階段，您將掌握最核心的幾個分析工具。

> **核心心態：** 從被動觀看轉變為主動調查，像偵探一樣尋找初步線索。

### 核心工具組

1.  `getSummary`：您的起點。快速了解這個 Jam 的大概內容。
2.  `getUserEvents`：重現使用者的完整操作路徑。
3.  `getConsoleLogs`：尋找前端的異常訊號。
4.  `getNetworkRequests`：檢查與後端的通訊是否正常。

### 基礎偵錯流程

這是一個萬用的基礎流程，可以解決 80% 的常見問題：

1.  **啟動分析**：對 Jam 連結使用 `getSummary`，閱讀 AI 給您的摘要和建議。
2.  **追蹤腳步**：使用 `getUserEvents`，一步步跟隨使用者的操作，了解他做了什麼。
3.  **檢查前端**：使用 `getConsoleLogs`，看看在使用者操作的過程中，瀏覽器是否拋出了任何錯誤或警告。
4.  **檢查後端**：使用 `getNetworkRequests`，特別是過濾 `statusCode` 為 `4xx` (客戶端錯誤) 或 `5xx` (伺服器錯誤) 的請求，看看 API 是否返回了錯誤。

**初學者任務**：使用一個已知的簡單 Bug 的 Jam（例如，一個壞掉的連結或一個表單驗證錯誤）。嚴格按照上述四個步驟進行分析，並嘗試僅使用這四個工具就定位到問題的根本原因。將您的分析過程記錄下來。

---

## 第三階段：偵探 (The Detective) - 串連線索

您現在已經可以處理簡單的 Bug。但現實世界中的問題往往更複雜，線索也更隱晦。在這個階段，您將學會如何處理視覺、語音和環境資訊，並將它們與您已有的線索串連起來。

> **核心心態：** 任何資訊都不是孤立的，時間戳是您最好的朋友。

### 進階工具組

*   `getScreenshot`：當 Jam 是以截圖形式提供時，這是您的主要視覺來源。
*   `getVideoTranscript`：聆聽使用者的心聲，他們在錄製時的口頭描述可能包含關鍵線索。
*   `getMetadata`：解開「環境之謎」。問題是只發生在特定版本、特定使用者，還是特定功能旗標開啟時？

### 跨工具分析技巧

*   **時間戳關聯**：在 `getVideoTranscript` 中聽到使用者說「啊，這裡錯了」的時間點，去 `getConsoleLogs` 和 `getNetworkRequests` 中尋找幾乎同一個時間戳的異常記錄。
*   **環境過濾**：從 `getMetadata` 中得知 `appVersion` 是 `2.1.0`。如果這個版本有一個已知問題，您的調查範圍就大大縮小了。
*   **視覺驗證**：在 `getUserEvents` 中看到使用者點擊了一個按鈕，但您不確定是哪個。使用 `getScreenshot` 或觀看影片來進行視覺確認。

**中級者任務**：挑戰一個更複雜的 Bug，例如一個只在特定條件下才會出現的渲染問題。您的任務是綜合運用第一、二階段的所有工具，並特別利用 `getMetadata` 來定義問題的邊界條件。最終，您需要產出一份報告，說明「這個 Bug 只在 App 版本為 X、使用者類型為 Y、且功能 Z 開啟時才會發生」。

## 第四階段：策略家 (The Strategist) - 自動化與規模化

作為一名偵探，您已經非常出色。但真正的大師不僅僅滿足於解決單一案件，他們會思考如何預防犯罪、如何管理整個轄區的治安。在這個階段，您將學會如何規模化地處理問題，並開始讓 AI 為您工作。

> **核心心態：** 從「解決一個 Bug」提升為「管理一類 Bug」，讓重複的工作自動化。

### 策略工具組

*   `searchJams`：您的雷達。從單一的 Jam 擴展到整個工作區，尋找問題的模式。
*   `analyzeVideo`：您的 AI 副手。讓 AI 為您進行初步分析，提取洞察。
*   `addComment` & `moveJam`：您的管理工具。自動化記錄結論和整理工作區。

### 自動化工作流程範例

想像一下，您可以建立一個 Manus Agent，它的工作流程如下：

1.  **監控**：定期使用 `searchJams` 尋找所有標題包含「緊急」且尚未處理的 Jam。
2.  **初步分析**：對每一個找到的 Jam，自動運行 `analyzeVideo` 和 `getSummary`。
3.  **分類**：如果 AI 分析報告中提到了「500 錯誤」，則自動使用 `addComment` 標記「#後端問題」，並 `moveJam` 到「後端待辦」資料夾。
4.  **通知**：如果 AI 分析報告中提到了「UI 渲染錯誤」，則自動在 Slack 中通知前端團隊。

**進階者任務**：設計一個您自己的自動化工作流程。這個流程的目標是：當一個新的 Jam 被建立時，您的 Agent 能夠自動判斷它的優先級（高、中、低），並將其分配給對應的團隊。您需要定義判斷優先級的規則（例如，包含「崩潰」、「無法登入」的為高優先級），並寫下完整的工具調用順序。

## 第五階段：建築師 (The Architect) - 建構偵錯生態

恭喜您，您已經到達了旅程的終點，同時也是一個新起點。在這個階段，您將超越工具本身，從系統和生態的角度來思考偵錯這件事。

> **核心心態：** Jam 不僅僅是一個工具，它是整個開發生命週期中的一個關鍵基礎設施。

### 大師級思維

*   **品質閘門**：將 Jam 分析整合到您的 CI/CD 流程中。例如，在部署到生產環境前，自動運行一個測試套件，如果產生了任何帶有嚴重錯誤的 Jam，則自動阻止部署。
*   **主動式監控**：建立一個 Agent，它不僅僅是等待 Bug 回報，而是主動使用 `searchJams` 分析過去一週的所有 Jam，尋找新出現的錯誤模式或趨勢，並生成一份「每週品質報告」。
*   **知識沉澱**：將 `addComment` 的使用制度化。要求所有分析過的 Jam 都必須有一個結論性的註解。這樣，Jam 不僅是 Bug 報告，更是一個可供檢索的偵錯知識庫。
*   **生態擴展**：思考您還能將 Jam 的數據與什麼系統整合？例如，將 `searchJams` 的結果與公司的財務數據關聯，分析「某個 Bug 模式造成了多少客戶流失或退款」。

**大師級挑戰**：設計一個「自我修復系統」的藍圖。這個藍圖需要描述一個理想的未來：一個 Jam 被建立後，一個 AI Agent 會自動分析它，定位到出錯的程式碼，生成修復方案，在沙箱環境中運行測試，最後提交一個 Pull Request 等待開發者審核。您的藍圖需要詳細說明這個過程中，Jam 連結器扮演了什麼角色，以及需要哪些其他工具（例如 GitHub MCP 連結器）來協同工作。

## 結語

從觀察者到建築師，您已經走過了一段漫長的旅程。您會發現，Jam 連結器不僅僅是關於「偵錯」，它更是關於「溝通」、「協作」和「系統化思考」。當您將這套哲學融入日常工作時，您釋放的不僅僅是個人的生產力，更是整個團隊的創新潛力。現在，開始您的旅程吧！
