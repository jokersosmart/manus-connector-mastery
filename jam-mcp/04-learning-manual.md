## 模組一：偵錯的「啊哈！」時刻

**目標：** 在 15 分鐘內，獨立完成一次從接收 Jam 到定位問題方向的基礎分析。

### 學習目標

*   **理解核心價值**：體會到擁有一個包含完整操作影片和日誌的「Jam 錄製檔」相比傳統文字報告的壓倒性優勢。
*   **掌握基礎工具**：學會使用 `getSummary` 快速掌握全貌，以及 `getUserEvents` 來精確追蹤使用者的每一步操作。

### 關鍵技能

1.  **`getSummary`**: 您的偵錯起點。它像一個經驗豐富的同事，直接告訴您這個 Jam 的重點和下一步該往哪裡查。
2.  **`getUserEvents`**: 您的「重播」按鈕。它讓您能以電腦的精度，而不是人腦的記憶，來重現使用者的操作流程。

### 實踐練習 1

*   **場景**：您收到一個 Jam 連結，回報說「網站上的『關於我們』連結壞了」。
*   **任務**：
    1.  對該 Jam 使用 `getSummary` 工具。
    2.  閱讀摘要後，使用 `getUserEvents` 工具，找出使用者具體點擊了哪個連結元素。
    3.  **回答問題**：根據工具的輸出，使用者點擊的連結 `href` 是什麼？它是否指向了錯誤的位址？

## 模組二：深入挖掘線索

**目標：** 能夠區分前端問題和後端問題，並使用正確的工具找到關鍵的錯誤訊息。

### 學習目標

*   **建立分層思維**：理解軟體問題可以大致分為瀏覽器端（前端）和伺服器端（後端）。
*   **掌握專業工具**：學會使用 `getConsoleLogs` 查找在瀏覽器中發生的 JavaScript 錯誤，以及使用 `getNetworkRequests` 來檢查與伺服器的通訊狀態。

### 關鍵技能

1.  **`getConsoleLogs`**: 您的前端聽診器。所有在使用者瀏覽器中發生的異常（紅色的 `error` 訊息是您的重點關注對象）都會在這裡現形。
2.  **`getNetworkRequests`**: 您的後端監視器。透過過濾 `statusCode`，您可以快速定位是使用者端請求錯誤 (`4xx`) 還是伺服器端處理錯誤 (`5xx`)。

### 實踐練習 2A：前端錯誤

*   **場景**：一個 Jam 回報「點擊『載入更多』按鈕後沒有任何反應」。
*   **任務**：
    1.  使用 `getConsoleLogs` 檢查日誌。
    2.  **回答問題**：您是否在日誌中找到類似 `TypeError` 或 `ReferenceError` 的錯誤？是哪個函式或變數出了問題？

### 實踐練習 2B：後端錯誤

*   **場景**：一個 Jam 回報「提交表單後，頁面顯示『伺服器錯誤』」。
*   **任務**：
    1.  使用 `getNetworkRequests` 並設定參數 `{"statusCode": "5xx"}`。
    2.  **回答問題**：出錯的 API 端點是哪一個？伺服器返回的具體錯誤訊息是什麼？

## 模組三：拼湊完整畫面

**目標：** 學會處理那些「只在特定情況下發生」的棘手問題，將環境因素納入考量。

### 學習目標

*   **理解上下文的重要性**：認識到 Bug 的發生往往與環境（如應用版本、使用者權限）和特定互動（如使用者的口頭描述）有關。
*   **掌握進階工具**：學會使用 `getMetadata` 讀取環境變數，`getVideoTranscript` 來「聽」懂使用者的意圖，以及 `getScreenshot` 進行視覺確認。

### 關鍵技能

1.  **`getMetadata`**: 您的環境掃描器。這是解決「在我電腦上是好的啊」問題的終極武器。檢查 `appVersion`, `userId`, `featureFlags` 等資訊。
2.  **`getVideoTranscript`**: 您的「讀心術」。使用者在錄製時的喃喃自語或清晰描述，往往包含了最直接的線索。
3.  **`getScreenshot`**: 您的像素級眼睛。當介面元素很難用文字描述時，一張截圖勝過千言萬語。

### 實踐練習 3

*   **場景**：一個 Jam 回報「一個只有 Beta 測試使用者才能看到的新功能按鈕，樣式錯亂了」。
*   **任務**：
    1.  使用 `getMetadata` 確認回報該 Jam 的使用者是否真的開啟了 `betaFeatures` 功能旗標。
    2.  使用 `getVideoTranscript` 聽取使用者對「樣式錯亂」的具體描述。
    3.  使用 `getScreenshot` 查看按鈕的實際渲染效果。
    4.  **回答問題**：綜合以上三點，請用一句話準確描述這個 Bug：「在 App 版本 X.X，開啟了 Y 功能旗標後，Z 按鈕出現了...的樣式問題」。

## 模組四：從單點到全面

**目標：** 培養從全局視角分析問題的能力，從處理單一 Bug 轉向識別和管理問題模式。

### 學習目標

*   **建立模式識別能力**：學會不只看單一事件，而是透過搜尋和匯總，發現重複出現的問題模式。
*   **學會授權 AI**：利用 `analyzeVideo` 等 AI 工具，將初步的、重複性的分析工作交給 AI，讓自己專注於更高層次的決策。

### 關鍵技能

1.  **`searchJams`**: 您的全域雷達。透過組合 `author`, `url` 等過濾條件，您可以找到所有相關的 Bug 報告，從而判斷問題的影響範圍和嚴重性。
2.  **`analyzeVideo`**: 您的 AI 分析師。在您深入研究之前，讓 AI 先幫您掃描一遍影片，找出潛在的異常點和使用者行為模式。

### 實踐練習 4

*   **場景**：老闆想知道「最近一週，我們的結帳頁面總共出現了多少問題？」
*   **任務**：
    1.  使用 `searchJams` 並設定參數 `{"url": "example.com/checkout"}`（假設結帳頁面 URL 包含此路徑），並自行加入時間範圍的限制（此處為概念性，實際工具可能需不同參數）。
    2.  對搜尋結果中的每一個 Jam，概念性地運行 `analyzeVideo`。
    3.  **回答問題**：根據搜尋和分析的結果，撰寫一個簡短的報告，總結結帳頁面最常出現的三個問題是什麼。
