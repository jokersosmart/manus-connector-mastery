### B. 收斂 (五何法)

#### 針對「自己 (開發者) - 過去 - Input control: 決定要暴露哪些 Supabase API 功能」

1.  **為什麼要決定暴露哪些功能？**
    > 因為 Supabase API 功能非常廣泛，全部暴露會導致 Connector 過於複雜，難以維護和使用。
2.  **為什麼全部暴露會過於複雜？**
    > 因為每個 API 端點都需要被包裝成一個獨立的 Manus 工具，過多的工具會讓使用者感到困惑，且開發工作量巨大。
3.  **為什麼過多的工具會讓使用者困惑？**
    > 因為使用者需要在數十甚至上百個工具中尋找他們需要的功能，學習曲線陡峭，降低了實用性。
4.  **為什麼陡峭的學習曲線會降低實用性？**
    > 因為 Manus 的核心價值之一是簡化操作，如果使用一個 Connector 比直接寫程式碼更複雜，它就失去了存在的意義。
5.  **為什麼失去存在意義？**
    > 因為沒有人會使用一個增加工作負擔而不是減少它的工具。
    > **根本原因：** 必須根據使用者的核心需求進行功能篩選，優先提供高價值、高頻率使用的 API，以確保 Connector 的實用性和易用性。

#### 針對「別人 (使用者) - 過去 - Input data: 現有工作流程中的痛點」

1.  **為什麼了解使用者的痛點很重要？**
    > 因為 Connector 的價值在於解決使用者的實際問題。不了解痛點，就無法設計出有用的功能。
2.  **為什麼要透過 Connector 解決痛點？**
    > 因為使用者希望在 Manus 的自然語言環境中，用簡單的指令完成複雜的操作，而不是手動呼叫 API 或編寫腳本。
3.  **為什麼使用者偏好自然語言指令？**
    > 因為這降低了技術門檻，讓非開發者也能利用 Supabase 的強大功能，實現業務流程自動化。
4.  **為什麼降低技術門檻很重要？**
    > 因為這能擴大 Supabase 和 Manus 的使用者基礎，讓更多人從中受益，創造更大的生態價值。
5.  **為什麼生態價值很重要？**
    > 因為一個活躍的生態能吸引更多開發者和使用者，形成正向循環，推動平台持續創新和成長。
    > **根本原因：** Connector 的成功取決於它能否精準地解決目標用戶在特定場景下的核心痛點，並將其無縫融入自然語言操作流程。

### 本輪根本原因清單

1.  **功能篩選：** 必須根據使用者的核心需求進行功能篩選，優先提供高價值、高頻率使用的 API，以確保 Connector 的實用性和易用性。
2.  **場景聚焦：** Connector 的成功取決於它能否精準地解決目標用戶在特定場景下的核心痛點，並將其無縫融入自然語言操作流程。
