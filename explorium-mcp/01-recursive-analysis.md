# 「Explorium」連結器遞迴分析

## 循環 Loop 1

### A. 發散 (4! 分析)

**思考模式：** 平行式

| 時間 | 元素 | 自己 (Manus) | 別人 (使用者) |
| :--- | :--- | :--- | :--- |
| **過去** | Input data | 現有的 MCP 框架、其他連結器開發經驗 | 使用者在其他平台整合外部數據的痛點 |

| | Library | 標準化的連結器開發流程、認證模組 | 使用者對數據擴充的既有知識 |

### B. 收斂 (五何法)

**元素 1: (自己, 過去, Input data) 現有的 MCP 框架、其他連結器開發經驗**

*   **核心概念：** 其他連結器開發經驗
*   **為什麼？** 因為這些經驗定義了我們開發新連結器的起點與限制。
*   **為什麼？** 因為標準化的框架能加速開發，但也可能限制了針對特定服務（如 Explorium）的深度整合潛力。
*   **為什麼？** 因為框架追求通用性，而 Explorium 提供的是高度專業化的數據擴充服務，兩者存在一定程度的矛盾。
*   **為什麼？** 因為通用性意味著犧牲部分客製化能力，可能無法完全發揮 Explorium 自動化特徵工程的優勢。
*   **根本原因：** 如何在通用 MCP 框架下，最大化發揮 Explorium 這類專業、自動化數據服務的獨特價值，是設計此連結器的核心挑戰。

**元素 2: (別人, 現在, Input data) 使用者的內部數據、具體的業務問題**

*   **核心概念：** 具體的業務問題
*   **為什麼？** 因為使用者不是為了用連結器而用，而是為了解決問題。
*   **為什麼？** 因為問題的定義（例如「提升客戶轉換率」或「降低模型預測偏差」）直接決定了需要擴充哪些外部數據。
*   **為什麼？** 因為 Explorium 的核心價值在於「自動」找到並整合最相關的外部數據來解決這些問題。
*   **為什麼？** 這意味著連結器的設計必須以「業務問題」為中心，而不是以「數據源」為中心。
*   **根本原因：** 連結器需要提供一種機制，讓使用者能用自然語言或簡單的參數描述他們的業務目標，而非要求他們成為外部數據專家。

### C. 本輪根本原因清單

1.  如何在通用 MCP 框架下，最大化發揮 Explorium 這類專業、自動化數據服務的獨特價值。
2.  連結器需要提供一種機制，讓使用者能用自然語言或簡單的參數描述他們的業務目標。
